## 1. Executive Summary

This RFC defines the architecture, scope, and interface contracts for the SYNTHRA Foundation Layer. The Foundation Layer is the set of core, low-level modules upon which all higher-level departments (Research, Knowledge, Execution, Learning, Portfolio, Infrastructure) are constructed.

By building a standardized foundation first, we ensure that subsequent modules (such as the Simulation Client, Experiment Database, and Agent Orchestrator) use consistent patterns for configuration, logging, exceptions, message passing, and dependency injection.

---

## 2. Problem Statement

Without a unified foundation, independent software modules will develop isolated implementations for basic services. This creates technical debt:
- Duplicate config parsing logic and environment variable reading.
- Inconsistent log formatting and output targets, complicating centralized debugging.
- Direct class coupling between layers, violating separation of concerns and preventing mock-based testing.
- Ad-hoc error propagation causing silent exceptions or unhandled crashes.

Establishing the Foundation Layer immediately resolves these risks by creating a unified environment before domain logic is implemented.

---

## 3. Objectives

- **Consistent Interface Design**: All settings, logs, and events must utilize shared, immutable models.
- **Decoupled Architecture**: Enforce dependency injection patterns to allow mocks and modules to be interchanged.
- **Fail Fast Initialization**: Validate all system variables and environment criteria during startup.
- **Comprehensive Testing Infrastructure**: Establish the testing harness (Pytest) and verification configurations.

---

## 4. Non-Objectives

The Foundation Layer will NOT:
- Implement direct quantitative research code, alpha generation rules, or operator libraries.
- Establish connections to the WorldQuant BRAIN ACE API.
- Manage database connections or vector memory persistence.
- Schedule campaign runs or manage LLM api transport.

---

## 5. Core Architectural Components

The Foundation Layer consists of the following conceptual modules:

```
+---------------------------------------------------------------+
|                      FOUNDATION LAYER                         |
+---------------------------------------------------------------+
|  1. Configuration System     |  2. Logging Framework          |
|  Type-safe load & validate   |  Centralized logging & traces  |
+------------------------------+--------------------------------+
|  3. Event Bus Interfaces     |  4. Dependency Injection       |
|  Publisher-Subscriber contracts |  Interface-based wiring     |
+------------------------------+--------------------------------+
|  5. Shared Types & Exceptions|  6. Plugin & Extension System  |
|  Core models & error trees   |  Runtime module loading        |
+---------------------------------------------------------------+
```

### 1. Configuration System
- **Responsibility**: Loads settings from config files (YAML/JSON) and system environments. Enforces validation rules at startup and outputs an immutable configuration container.

### 2. Logging Framework
- **Responsibility**: Provides formatted output streams (stdout, local log files) with explicit severity levels and contextual campaign metadata tags.

### 3. Event Bus Interfaces
- **Responsibility**: Defines the abstract contracts (publishers, subscribers, message containers) enabling asynchronous, decoupled messaging.

### 4. Dependency Injection Container
- **Responsibility**: Manages instance lifecycles and handles constructor injection based on interface specifications.

### 5. Plugin System
- **Responsibility**: Provides hooks to locate, validate, and load external components at runtime using standard directory configurations.

### 6. Shared Types & Exceptions
- **Responsibility**: Defines base exceptions and core domain models (such as Campaigns, Hypotheses, and Experiments metadata structures).

## 6. Implementation Plan & Deliverables

Once this RFC is approved, we will construct the following specifications and codebases sequentially:

1.  **SPEC-0001: Configuration Manager**: Loader, parser, validator, and configuration schemas.
2.  **SPEC-0002: Logging & Diagnostics**: Logger configuration, formatters, and output handlers.
3.  **SPEC-0003: Core Exception Hierarchy**: Central error definitions and base class.
4.  **SPEC-0004: Event Bus**: InMemory implementation of event distribution.
5.  **SPEC-0005: Dependency Injector**: Lightweight type-based constructor injection.
6.  **SPEC-0006: Plugin Loader**: Dynamic module lookup and registry checks.
