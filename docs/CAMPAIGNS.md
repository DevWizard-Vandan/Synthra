# SYNTHRA Campaigns Specification

This document defines the Campaign lifecycle within SYNTHRA. In SYNTHRA, all research tasks are organized under a centralized Campaign object, which serves as the primary unit of research.

---

## 🏛️ The Campaign as a Repository

A Campaign is the research equivalent of a Git repository. It is the primary bounding context for all operations. No hypothesis is written, no expression is synthesized, and no simulation is run outside of an active Campaign.

```
                         CAMPAIGN (The Unit of Research)
                            │
         ┌──────────────────┼──────────────────┐
         ▼                  ▼                  ▼
    Hypotheses         Experiments      Research Assets
    (HYP-XXXX)          (EXP-XXXX)         (AST-XXXX)
                            │
                            v
                        Knowledge (compounded database updates)
                            │
                            v
                  Submission Candidates
```

Every database entry, vector embedding, and source file generated by SYNTHRA is tagged with a Campaign ID (`CMP-XXXX`). This ensures that:
- **Lineage is Traceable**: Every strategy candidate can be traced back to its campaign parameters, economic themes, and underlying hypothesis.
- **Resource Isolation**: Computational budgets, API limits, and token caps are enforced at the Campaign boundary.
- **Incremental Knowledge**: Campaigns compile structured summaries of what was learned, adding to the Knowledge Graph rather than generating single, isolated alphas.

---

## 🔄 Campaign Lifecycle Phases

### 1. Campaign Formulation (`CMP-XXXX: Draft`)
- **Action**: The user or planning core defines a Campaign scope.
- **Parameters**: Specifies the geographic region, universe size, holding periods, and the target economic theme.
- **Objective**: Establish the bounds within which the system is authorized to run tests.

### 2. Inquiry Formulation (`HYP-XXXX: Draft`)
- **Question**: The Research Department formulates specific scientific inquiries matching the campaign theme.
- **Hypothesis**: The system generates testable economic statements explaining the target relationship.

### 3. Data & Operator Mapping
- **Datasets**: The system selects specific fields from the platform catalog that represent the variables in the Hypothesis.
- **Operators**: The system filters the platform operator set, selecting mathematical and statistical transformations matching the hypothesis structure.

### 4. Strategy Generation & Execution (`EXP-XXXX: Running`)
- **Variants**: The synthesizer compiles multiple code variations (`AST-XXXX`) representing different combinations of variables and operators.
- **Simulations**: The execution layer schedules backtests for all variants via the ACE API and collects raw outcomes.

### 5. Learning & Retrospective (`CMP-XXXX: Concluded`)
- **Learning**: The learning department evaluates all simulation outputs, grouping errors, compiling correlations, and updating the Knowledge Graph.
- **Conclusion**: The campaign terminates when the predefined budget (token limit, simulation count, or time) is exhausted. The system outputs a campaign conclusion report summarizing the knowledge gained and preparing strategy candidates for human review.
